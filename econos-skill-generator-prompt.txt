## CONTEXT: What econos is

econos is a static educational web app for A-Level Economics. It has exactly six HTML shells at the repo root:

```
index.html  learn.html  link.html  land.html  quiz.html  login.html
```

Topics live in `js/data/<topic-id>/data-topic.js`. No new HTML files are ever added. The shells discover topics through `TopicLoader.loadData(...)` at runtime.

A topic is registered in `js/topics.js` with fields including `id`, `title`, `theme`, `unit`, `order`, and a short tag list — match the style of surrounding entries exactly.

Topic IDs use **underscores**, not hyphens. Examples: `factors_of_production`, `price_controls`, `growth_measure`.

### Important: SVG diagrams

SVGs are **never inlined into card body fields**. They live as named keys in `js/icons.js` (as `ECONOS_ICONS.keyName = \`<svg...>\``) and are referenced by cards via `visualKey: 'keyName'` or `diagramKey: 'keyName'`. The `econos-visual-diagram` skill governs SVG authoring. When GPT-4o extraction describes a diagram in a body field, the skill must convert it to a named SVG in `js/icons.js` and replace the body text with `visualKey: 'keyName'`. SVG viewBox is always `viewBox="0 0 640 H"` where H is the actual pixel height — width is fixed at 640, height varies by content.

### Tone values

The six valid tones: `green`, `amber`, `blue`, `purple`, `rose`, `slate`. These map to fixed background colours in the renderer. Do not invent other values.

### Steps (card-level, not flow tiles)

Some cards use a top-level `steps` array for ordered narrative steps:

```js
steps: [
  { label: 'Step heading', text: 'Step explanation.' }
]
```

This is distinct from the `flow` array of tiles.

## CONTEXT: GPT Image 2 API

The skill uses GPT Image 2 to generate mockup images for each card. Use this exact API shape — no deviations:

```python
import openai, base64, os, sys

api_key = os.environ.get("OPENAI_API_KEY")
if not api_key:
    sys.exit("ERROR: OPENAI_API_KEY is not set in the environment. Add it to .claude/settings.local.json under the 'env' block.")

client = openai.OpenAI(api_key=api_key)

response = client.images.generate(
    model="gpt-image-2-2026-04-21",   # pinned snapshot — never use rolling alias "gpt-image-2"
    prompt="...",
    n=1,
    size="1024x1024",                  # valid: 1024x1024 | 1024x1792 | 1792x1024
    quality="high",                    # valid: standard | high
    output_format="b64_json"           # always request base64
)

image_bytes = base64.b64decode(response.data[0].b64_json)
with open("card_01_mockup.png", "wb") as f:
    f.write(image_bytes)
```

**There is no `style` parameter** — omit it entirely or the call will be rejected.

The `OPENAI_API_KEY` lives in `.claude/settings.local.json` under the `env` block and is surfaced to Claude Code automatically. It is never committed to the repo. The script must check for its presence and exit with a clear message if absent.

Pin the snapshot ID `gpt-image-2-2026-04-21` rather than the rolling alias `gpt-image-2` so a future model bump does not silently change image style mid-batch.

## CONTEXT: Full build-from-mockup workflow

The workflow has five stages:

**Stage 1 — Topic brief**
Receive from the user: topic ID (underscore_case), title, approximate card count, and the economic concept. Load the `econos-card-template` skill for field reference.

**Stage 2 — Generate mockup images with GPT Image 2**
For each card, call the GPT Image 2 API (see API shape above). The image prompt must describe:
- The card's pedagogical purpose (e.g. "an infographic card explaining the determinants of PED for A-Level Economics students")
- Visual style: clean infographic, white background, econos colour palette (muted pastels: green #D1FAE5, blue #DBEAFE, amber #FEF3C7, purple #EDE9FE, rose #FFE4E6, slate #F1F5F9), clear typography, no decorative clutter
- Any diagrams (e.g. supply-demand curves, PPF, AD/AS — described in plain terms)
- Key headings, labels, and icons that should appear

Save each image as `mockup_card_NN.png` in `.claude/skills/econos-build-from-mockup/tmp/<topic_id>/`.

**Stage 3 — Extract structured content from images**
Send each mockup PNG to the vision model as a base64-encoded image with this extraction prompt:

```
You are extracting structured content from an econos card mockup image for an A-Level Economics web app.
Return ONLY valid JSON. Use this exact shape:

{
  "id": "topic_id_N",
  "template": "ad-interactive",
  "stepLabel": "Learn: Step N of M",
  "title": "Card title",
  "lede": "Opening paragraph.",
  "tip": null,
  "left": null,
  "right": null,
  "flow": [],
  "causes": [],
  "keyTerms": [],
  "conclusion": null,
  "exam": null,
  "diagramDescription": "Plain-text description of any diagram shown, or null if none."
}

Rules:
- Only include array/object fields that have content. Omit empty arrays and null fields from output.
- causes items: { "tone": "blue|green|amber|purple|rose|slate", "icon": "emoji", "head": "heading", "body": "body text" }
- flow items: { "tone": "...", "icon": "emoji", "title": "step title", "sub": "explanation" }
- If a diagram is shown, describe it in diagramDescription — Claude will convert this to SVG later. Do not attempt to produce SVG here.
- tone must be one of: green, amber, blue, purple, rose, slate
- All icon fields must be a single emoji character
- Be economically accurate — this is A-Level Economics content
```

Parse the returned JSON. Flag any card where parsing fails for manual review. Assemble all cards into a draft `data-topic.js` using the schema above.

**Stage 4 — Claude review, SVG conversion, and validation**
Read the draft and:
- Correct any extraction errors: wrong tones, garbled text, misread icons, economically inaccurate statements
- For each card with a `diagramDescription`: load the `econos-visual-diagram` skill, convert the description to a named SVG in `js/icons.js`, replace `diagramDescription` with `visualKey: 'keyName'`
- Ensure the file uses `window.ECONOS_TOPIC_DATA['<id>'] = {...}` — not `const topicData` or any other shape
- Validate: `node --check js/data/<topic-id>/data-topic.js` — must pass with no errors
- Validate: `node --check js/icons.js` — must pass after any SVG additions

**Stage 5 — Register, verify, and ship**
1. Add the topic to `js/topics.js` — load the `econos-new-topic` skill for the exact registration shape.
2. Run `scripts/lint.sh` — must pass.
3. Load the `run-econos` skill — take a screenshot of the topic at card index 0, then navigate through all cards to confirm they render without errors.
4. **Stop here.** Hand off to the `econos-ship-changes` skill for the push → PR → squash-merge → dev-branch reset sequence. Do not duplicate that logic in this skill.
